1 HACL*
2 =====
3 
4 HACL* is a formally verified cryptographic library in [F\*],
5 developed by the [Prosecco](http://prosecco.inria.fr) team at
6 [INRIA Paris](https://www.inria.fr/en/centre/paris) in collaboration
7 with Microsoft Research, as part of [Project Everest].
8 
9 HACL stands for High-Assurance Cryptographic Library and its design is
10 inspired by discussions at the [HACS series of workshops](https://github.com/HACS-workshop).
11 The goal of this library is to develop verified C reference implementations
12 for popular cryptographic primitives and to verify them for memory safety,
13 functional correctness, and secret independence.
14 
15 More details about the HACL* library and its design can be found in our ACM CCS 2017 research paper:
16 https://eprint.iacr.org/2017/536
17 
18 All our code is written and verified in [F\*] and then compiled to C via
19 the [KreMLin tool](https://github.com/FStarLang/kremlin/). Details on the verification and compilation
20 toolchain and their formal guarantees can be found in the ICFP 2017 paper:
21 https://arxiv.org/abs/1703.00053
22 
23 # Supported Cryptographic Algorithms
24 
25 The primitives and constructions supported currently are:
26 
27 * Stream ciphers: Chacha20, Salsa20, XSalsa20
28 * MACs: Poly1305, HMAC
29 * Elliptic Curves: Curve25519
30 * Elliptic Curves Signatures: Ed25519
31 * Hash functions: SHA2 (256,384,512)
32 * NaCl API: secret_box, box, sign
33 * TLS API: IETF Chacha20Poly1305 AEAD
34 
35 Developers can use HACL* through the [NaCl API].
36 In particular, we implement the same C API as [libsodium] for the
37 NaCl constructions, so any application that relies on
38 libsodium only for these constructions can be immediately ported to use the verified code in HACL*
39 instead. (Warning: libsodium also implements other algorithms not in NaCl
40 that are not implemented by HACL*)
41 
42 The verified primitives can also be used to support larger F* verification projects.
43 For example, HACL* code is used through the agile cryptographic model developed in
44 [secure_api/] as the basis for cryptographic proofs of the TLS record layer in [miTLS].
45 A detailed description of the code in [secure_api/] and its formal security guarantees
46 appears in the IEEE S&P 2017 paper: https://eprint.iacr.org/2016/1178.pdf

84 # Verifying and Building HACL*
85 
86 Type `make` to get more information:
87 ```
88 HACL* Makefile:
89 If you want to run and test the C library:
90 - 'make build' will generate a shared library from the hacl-c snapshot (no verification)
91 - 'make unit-tests' will run tests on the library built rom the hacl-c snapshot (no verification)
92 - 'make clean-build' will clean 'build' artifacts
93 
94 If you want to verify the F* code and regenerate the C library:
95 - 'make prepare' will try to install F* and Kremlin (still has some prerequisites)
96 - 'make verify' will run F* verification on all specs, code and secure-api directories
97 - 'make extract' will generate all the C code into a snapshot and test it (no verification)
98 - 'make test-all' will generate and test everything (no verification)
99 - 'make world' will run everything (except make prepare)
100 - 'make clean' will remove all artifacts created by other targets
101 ```
102 
103 Verification and C code generation requires [F\*] and [KreMLin].
104 Benchmarking performance in `test-all` requires [openssl] and [libsodium].
105 An additional CMake build is available and can be run with `make build-cmake`.

108 # Performance
109 
110 To measure see the performance of HACL* primitives on your platform and C compiler,
111 run the targets from `test/Makefile` if you have the dependencies installed. (experimental)
112 To compare its performance with the C reference code (not the assembly versions) in [libsodium] and [openssl],
113 download and compile [libsodium] with the `--disable-asm` flag and [openssl] with the `-no-asm` flag.
114 
115 While HACL* is typically as fast as hand-written C code, it is typically 1.1-5.7x slower than
116 assembly code in our experiments. In the future, we hope to close this gap by using verified assembly implementations
117 like [Vale](https://github.com/project-everest/vale) for some primitives.
118

120 [libsodium]: https://github.com/jedisct1/libsodium

123 # Experimental features
124 
125 The [code/experimental](code/experimental) directory includes other (partially verified) cryptographic primitives that will become part of the library in the near future:
126 * Encryption: AES-128, AES-256
127 * MACs: GCM
128 * Key Derivation: HKDF
129 * Signatures: RSA-PSS
130 
131 We are also working on a JavaScript backend for F* that would enable us to extract HACL* as a JavaScript library.
